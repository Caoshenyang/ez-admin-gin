# 文档编写规范

这份规范用于指导 `EZ Admin Gin` 仓库中的 Markdown / VitePress 文档写作。它不是排版清单，而是一套帮助读者继续读下去的写作设计：先给清晰结论，再降低理解成本，最后让读者知道下一步怎么验证。

::: tip 这份规范的目标
好文档要做到三件事：让读者知道这页值不值得读、让关键操作一眼能找到、让每一步都有可验证结果。
:::

::: info 关于页面导航
如果页面右侧已经显示 VitePress 大纲，正文里通常不需要再重复放 `## 页面导航` 和 `[[toc]]`。

只有当页面关闭了右侧大纲，或页面本身特别依赖正文内目录时，才建议使用 `[[toc]]`。
:::

## 北极星原则

| 原则 | 要做到什么 | 常见反例 |
| --- | --- | --- |
| 先给答案 | 开头 30 秒内让读者知道这页解决什么问题 | 一上来先讲很长背景 |
| 一屏一重点 | 每个小节只解决一个核心问题 | 一个小节同时讲概念、命令、排错和部署 |
| 让读者少猜 | 明确指出前置条件、输入输出和验收方式 | 默认读者知道上下文 |
| 让页面有节奏 | 用标题、折叠块、代码组和提示块切分阅读负担 | 整页连续文本或巨型代码墙 |
| 先可用，再完整 | 优先给最小可运行路径，再补原理和细节 | 把最重要的操作埋在后面 |

> [!IMPORTANT]
> 教程页不是论文，也不是 API dump。读者愿不愿意继续学，往往取决于前 20 秒有没有看到“我现在就能做成”的信号。

## 页面骨架模板

### 教程页

教程页的读者目标是“照着做并完成验证”。推荐结构：

```md
---
title: 页面标题
description: "一句话说明这页帮助读者完成什么。"
---

# 页面标题

这一页会带你完成什么，完成后你能看到什么结果。

::: tip 阅读建议
第一次先照着主线跑通；第二遍再回来看 details 里的原理和扩展。
:::

## 你会完成什么

- 目标 1
- 目标 2
- 可验证结果

## 前置条件

- 已完成哪些步骤
- 需要哪些工具或文件

## 第一步：做一件明确的事

给出命令、配置或操作。

## 验证结果

说明应该看到什么输出、文件或页面。

::: warning 易错点
指出最容易失败的位置，以及如何判断。
:::

## 小结

用 3-5 条短句收束，并告诉读者下一页做什么。
```

### 参考页

参考页的读者目标是“快速查答案”。推荐结构：

```md
---
title: 配置参考
description: "列出某类配置项、默认值、示例和注意事项。"
---

# 配置参考

这页用于快速查阅某类配置。

## 快速索引

| 配置 | 用途 | 默认值 |
| --- | --- | --- |
| `APP_ENV` | 运行环境 | `dev` |

## 配置项

### `APP_ENV`

- 类型：字符串
- 示例：`dev` / `prod`
- 说明：控制运行环境。
```

### FAQ 页

FAQ 页适合把问题单独放进 `details`，避免读者被一长串答案压住。

```md
---
title: 常见问题
description: "收集本阶段最常见的问题和解决方式。"
---

# 常见问题

::: details 为什么命令执行失败？
先检查工作目录，再检查依赖是否安装。
:::

::: details Windows 和 macOS 的命令一样吗？
如果存在差异，使用 code-group 分开写。
:::
```

## Markdown 扩展怎么用

### Frontmatter

每一页都必须至少包含：

- `title`
- `description`

写法建议：

- `title` 使用读者会搜索的关键词。
- `description` 用一句话说明“这页解决什么问题”。
- 不要把 `description` 写成口号，应该写成可判断的结果。

### 标题和目录

- 二级标题负责导航，三级标题负责拆解动作。
- 不要出现“一个二级标题下面塞 800 行内容”。
- 长文不是默认都要加 `[[toc]]`。
- 如果右侧大纲已经可用，优先只保留右侧大纲。
- 只有在页面关闭右侧导航，或正文内目录能明显降低跳转成本时，才添加 `[[toc]]`。

::: details 什么时候可以考虑加正文目录
- 页面内容超过 6 屏。
- 同时存在多个步骤或多个子主题。
- 页面未来会被频繁回查，而不是只读一次。
- 右侧大纲不可见或已显式关闭。
:::

### 提示块

GitHub Alerts 和 VitePress 容器都可以使用，但要分工明确。

> [!TIP]
> 用于推荐做法、阅读建议和记忆口诀。

> [!NOTE]
> 用于补充背景、上下文关系和非主线说明。

> [!WARNING]
> 用于高频踩坑、生产风险、路径差异和版本差异。

::: info 使用建议
如果想要中文标题和更统一的视觉语气，优先用 VitePress 的 `tip / info / warning / details` 容器；如果想做更醒目的编辑提醒，可以使用 GitHub Alerts。
:::

### 表格

表格适合做并列比较，不适合塞长段解释。

推荐用于：

- 方案对比
- 参数说明
- 错误与解决方式
- Go vs Java 对照
- 阶段验收标准

### 代码块

代码块必须满足三个标准：

1. 标记正确语言。
2. 默认给最小可运行示例。
3. 需要时标出重点行，而不是让读者自己找。

行高亮示例：

```go{2,5-7}
func NewServer() *http.Server {
	engine := gin.Default()
	return &http.Server{
		Addr:         ":8080",
		Handler:      engine,
		ReadTimeout:  30 * time.Second,
		WriteTimeout: 30 * time.Second,
	}
}
```

代码聚焦和差异示例：

```ts
const columns = [
  { title: '用户名', key: 'username' }, // [!code ++]
  { title: '邮箱', key: 'email' }, // [!code focus]
  { title: '手机号', key: 'phone' } // [!code --]
]

submitForm(payload) // [!code warning]
unsafeEval(payload) // [!code error]
```

### 代码组

多平台、多包管理器、多调用方式时使用 `code-group`，不要把不同命令硬塞进同一个代码块。

::: code-group

```bash [pnpm]
pnpm install
pnpm docs:dev
```

```bash [npm]
npm install
npm run docs:dev
```

```bash [yarn]
yarn
yarn docs:dev
```

:::

### 折叠块

`details` 不是为了藏内容，而是为了控制阅读节奏。

适合折叠：

- 可选方案
- 深入原理
- 平台差异
- 长排错清单
- 不影响主流程的补充说明

不适合折叠：

- 本页结论
- 第一条可运行命令
- 核心配置
- 验收标准

### 导入片段

复用高频片段时，可以使用 VitePress 的代码导入或 Markdown include。

```md
<<< @/snippets/example.http

<!--@include: ./_includes/checklist.md-->
```

使用前要确认被导入文件真实存在。不要为了演示语法而留下无法构建的路径。

## 怎么把页面写得更好读

### 开头先给继续读下去的理由

推荐开头结构：

1. 这页解决什么痛点。
2. 读完能做成什么。
3. 建议怎么读或怎么验证。

### 让文字有镜头感

可以多用这些写法：

- “先别急着记住全部细节，先抓住这个判断标准。”
- “如果你只记一件事，就记这一条。”
- “这一步看起来小，但会决定后面几章顺不顺。”
- “第一次读先照着跑，第二遍再回来看原理。”

### 克制使用 Emoji

Emoji 可以作为视觉路标，但不能成为装饰噪音。

推荐语义：

| Emoji | 适合表达什么 | 推荐场景 |
| --- | --- | --- |
| `🧭` | 导航、路线、章节入口 | 课程大纲、阅读建议 |
| `🎯` | 目标、收益、结果 | 学习目标、先看结论 |
| `🚀` | 开始动手、快速启动 | 快速开始、第一步操作 |
| `🛠️` | 实操、配置、命令 | 环境要求、安装步骤 |
| `💡` | 技巧、提醒、理解要点 | 阅读建议、关键理解 |
| `⚠️` | 风险、坑点、注意事项 | warning、易错点 |
| `✅` | 完成、推荐、正确做法 | 检查清单、最佳实践 |
| `📌` | 重点、结论、记忆点 | 小结、本文重点 |

::: details 一个安全的默认原则
如果不确定某个 Emoji 会不会显得太多，就问自己一句：去掉它之后，这一段会不会更难扫读？

- 如果答案是“不会”，删掉。
- 如果答案是“会”，保留。
:::

## 禁忌清单

::: danger 千万别这样写
- 标题全是抽象名词，比如“实现原理分析”。
- 一上来先贴 150 行代码，没有一句导读。
- 多平台命令写在一个代码块里让读者自己猜。
- 易错点埋在正文中部，没有任何提示块。
- 同样的说明在多个页面里手动复制。
- 为了炫技乱用扩展，反而增加理解成本。
- 把主流程和支线内容混在一起写。
:::

## 发布前检查

::: details 文档发布前 10 项检查表
- 页面是否有清晰的 `title` 和 `description`。
- 开头是否在 30 秒内告诉读者“这页值不值得读”。
- 如果右侧大纲关闭，长文是否补了 `[[toc]]`。
- 是否至少有一处真正帮助读者避坑的提示块。
- 多平台命令是否用了 `code-group`。
- 关键代码是否做了行高亮或聚焦。
- 可复用内容是否抽成代码片段或 include。
- 链接是否都能跳转。
- 术语是否统一。
- `pnpm docs:build` 是否通过。
:::
