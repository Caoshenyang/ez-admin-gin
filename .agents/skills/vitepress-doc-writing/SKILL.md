# VitePress 文档写作

为本仓库编写和编辑文档时，只追求一个目标：让下一位读者愿意继续读下去。

## 什么时候使用这个 skill

当任务涉及以下内容时，使用这个 skill：

- 编写或重写 `docs/` 下的 Markdown 页面
- 优化教程的可读性、结构和阅读节奏
- 统一 VitePress 文档风格
- 补充示例、警告块、代码组、目录或其他 Markdown 扩展
- 把密集、偏硬的技术说明改写成更适合初学者阅读的学习材料

## 开始前先读的文件

在动手前，先阅读这个 skill 自带的两个参考文件：

1. `references/documentation-writing-standard.md`
2. `references/markdown-enhancements.md`

第一个文件是写作标准，第二个文件是 Markdown 扩展语法手册。

## 默认工作流

1. 先判断页面类型：教程页、参考页、FAQ，还是规划页。
2. 先让首屏有价值：
   - 标题清晰
   - 一句话讲明白这页解决什么问题
   - 需要时补一个 `tip` 或 GitHub alert 说明“这页怎么读”
3. 默认不要在正文里补 `## 页面导航` 和 `[[toc]]`：
   - 如果页面右侧已经有 VitePress 大纲（`outline` / `aside` 正常显示），正文里再放一份目录通常是重复的
   - 只有在页面显式关闭了右侧大纲，或页面类型本身强依赖正文目录（如 FAQ、特殊汇总页）时，才考虑使用 `[[toc]]`
4. 重新组织内容节奏：
   - 先讲结论
   - 再给例子
   - 再解释原理
   - 再提示风险或易错点
   - 最后给下一步
5. 有意识地使用 Markdown 扩展：
   - `tip / info / warning / details`
   - GitHub alerts，用于更强的编辑提醒
   - `code-group`，用于平台或包管理器差异
   - 行高亮，用于强调关键代码
   - `focus / ++ / -- / warning / error` 代码标记，用于提升扫读效率
   - 对重复内容使用导入代码片段或 Markdown include
   - 在确实能增强扫读体验时，使用少量语义稳定的 Emoji 作为视觉路标
6. 教程型页面优先使用 `tip / warning / details / code-group` 这些稳定语法：
   - `tip` 放本节目标、阅读建议或关键判断
   - `warning` 放真实易错点，不放泛泛提醒
   - `details` 放可选背景、版本差异、补充解释
   - `code-group` 放平台差异命令
   - Emoji 只作为少量视觉路标，优先用于目标、操作、风险、验证
7. 如果有助于阅读推进，在结尾加一个简短小结、检查表或下一页过渡。

## 按页面类型处理

### 教程页

- 开头先告诉读者读完能做成什么。
- 先给最小可运行路径，再讲深层原理。
- 默认至少安排一个 `tip` 或 `warning` 来托住阅读节奏。
- 用 `details` 把“主线内容”和“可选深入内容”分开。
- 至少补一处高价值警告，提醒最容易踩坑的点。

### 参考页

- 优先服务“快速查阅”，不是讲故事。
- 多用分组列表、简洁表格和短示例。
- 不要把答案埋在大段正文里。

### FAQ 页

- 仅在右侧大纲被关闭、或确实需要正文内目录时使用 `[[toc]]`。
- 每个问题尽量单独放进一个 `details`。
- 存在平台差异时，用 `code-group` 包住命令。

## 本仓库的写作规则

- 以中文为主。
- 语气保持温和、务实、能增强读者信心。
- 在确实有帮助时，加入 “Go vs Java” 的对照视角。
- 站内链接使用无扩展写法。
- 示例尽量真实、可运行。
- 不要为了“功能多”而乱加扩展；每一个扩展都必须降低读者理解成本。
- Emoji 可以使用，但必须克制、稳定、有语义，优先作为目标、操作、风险、验证的路标，不要让它们盖过内容本身。
- 如果页面右侧已经有大纲，默认不要再在正文里重复放“页面导航”。

## 本项目专属要求

- 只负责编写、审阅和修正文档，不直接生成后端、前端或部署实现代码。
- 文档必须服务于用户手动验证：关键步骤要写清前置条件、操作、期望结果和失败排查。
- 修改已有文件时，先用短列表说明“本次要改哪些点”，再给代码片段或完整代码，避免读者不知道改动边界。
- 修改已有文件的短代码片段，优先使用 VitePress 的 `// [!code ++]`、`// [!code --]`、`// [!code focus]` 标出新增、删除和重点行；YAML / Shell 等语言可使用对应注释语法。完整新增文件不需要整段标绿。
- 修改 Go 文件并新增 import 时，要提醒读者 IDE 可能自动删除暂时未使用的 import；推荐一次性替换完整文件，或先粘贴使用代码，再让 IDE 自动补全 / 整理 import。这里要拆成两层：先用 `warning` 说明“这是正常现象，不是学员操作错了”；再用 `details` 回答“怎么关闭 / 怎么处理”，并说明是否关闭由读者自行决定。
- 如果某个提示是跨章节通用的编辑器或工具行为，例如 GoLand 自动整理 import、终端代理、编码格式等，优先集中放在教程总入口或统一 FAQ，不要散落到每一篇正文里重复出现。
- 安装新依赖时，紧跟依赖用途和资料入口；Go 依赖优先给 `pkg.go.dev`，必要时再补项目仓库或官网。
- 教程代码块要补关键注释：解释结构、意图、风险点和验证点；不要逐行注释常识性语法，也不要把注释写成噪音。
- 写作主线始终围绕“EZ Admin Gin 是面向个人项目快速上线的通用后台管理系统底座”。
- 当文档涉及具体命令、目录、配置或版本时，优先保证准确性，其次才是表达吸引力。
- 如果某一步需要用户实际执行验证，文档要明确说明“执行后应该看到什么”，避免只写操作不写验收。
- 正文只写给学习者看的内容，不把“编写者的心里话”或内部写作提醒写进页面。例如不要写“不要把它写成普通教程”“后续每一章都应该服务这个目标”这类编辑指令；应改写成学习者能获得的知识、判断标准或操作建议。
- 技术表述保持简洁：正文默认写“通用后台管理系统底座”或“后台底座”，不要反复写 `Go/Gin` 或 `Gin`。仓库名、首页、技术栈说明、具体框架接入步骤中可以出现 `Gin`；只有在必须强调语言和框架组合时，才写成 `Go + Gin`。
- 保持教学者视角：`docs` 是教案和教学载体，但正文不要向学员暴露“我们正在编写文档、验证文档、后续修正文档”的幕后过程。需要提到文档时，只把它作为项目交付物、使用说明或学习资料的一部分来讲。

## 完成前的质量门槛

提交前检查：

- 页面是否有 `title` 和 `description`
- 开头是否解释了“这页为什么值得读”
- 密集内容是否被标题或容器拆开
- 代码示例是否标了正确语言
- 关键操作是否一眼能找到
- 重复内容是否被抽取，而不是手动复制
- 关键步骤是否包含前置条件、操作、期望结果和失败排查
- 修改已有文件前是否说明了改动点
- 修改片段是否用 diff / focus 标记突出新增或重点行
- 新增 Go import 的步骤是否提示了 IDE 自动整理 import 的影响
- 新增依赖是否给出用途和资料入口
- 代码块是否有必要的关键注释，且没有过度注释
- 是否清理了面向编写者的内部提醒，只保留面向学习者的说明
- 是否避免暴露教案编写、文档验证、后续修订等幕后过程
- 进行过实质性文档修改后，`cd docs; pnpm docs:build` 是否通过

## 默认优先动作

- 教程页顶部补一个短 `tip`，说明本节做成什么、怎么验证。
- 把长段风险提示改成 `warning` 容器
- 把可选深入内容改成 `details`
- 把平台差异命令改成 `code-group`
- 仅在没有右侧目录时，给长文补 `[[toc]]`
- 在页面顶部补一句“这页怎么读”
- 在高层标题或关键提示处，补少量能增强扫读感的 Emoji
- 给验证型步骤补“执行后应该看到什么”
- 给新增依赖补用途说明和官方资料入口
- 给复杂代码块补少量关键注释
- 给修改已有文件的步骤补“本次改动点”
- 给短修改片段补 `// [!code ++]` / `// [!code --]` / `// [!code focus]`，提升差异识别速度
- 给新增 Go import 的步骤补 GoLand / IDE 自动整理 import 提醒

## 反模式

- 巨型文本墙
- 右侧已有目录，却在正文里再重复一遍“页面导航”
- 只贴代码，不解释读者为什么要关心
- 使用像“实现原理分析”这种抽象标题，却不给结果信号
- 把主流程和支线内容混在一起写
- 同一条说明在多个页面里手动重复
- 为了好看而堆格式，反而增加认知负担
- 把 Emoji 当成装饰贴纸乱用，导致页面显得轻浮或杂乱
- 只写命令，不写执行位置、预期输出和失败时怎么判断
- 把写作策略、编辑提醒或作者自我约束直接写给读者看
- 把 `docs` 讲成编写者自己的工作区，而不是学员可阅读的项目文档和学习资料
- 修改文件时直接贴整段代码，却不说明改了哪里
- 修改片段没有使用 diff 或 focus 标记，导致读者仍然要自己找新增行
- 让读者单独粘贴 Go import，却不提醒 IDE 可能会删除未使用 import
- 安装依赖只给命令，不给资料入口
- 代码块完全没有注释，读者只能靠猜理解关键意图
- 为了“有注释”给每一行都写废话注释
