# 用户模型与登录

认证主线的第一步，不是先讲 JWT，而是先让系统知道“谁可以登录”。这一节先把用户模型、首次管理员初始化和用户名密码登录入口放到同一条线上，让后续的 Token、菜单和权限判断都有稳定起点。

::: tip 🎯 本节目标
完成后，系统会具备 `sys_user` 账号模型、一次性管理员初始化入口，以及 `/api/v1/auth/login` 登录接口。登录成功时会进入最终版认证模块，而不是停留在临时 Handler 拼装阶段。
:::

## 先说明当前边界

当前主线里，认证起步拆成两件事：

| 能力 | 作用 |
| --- | --- |
| `/api/v1/setup/init` | 创建首个管理员账号，并绑定到 `super_admin` |
| `/api/v1/auth/login` | 校验用户名密码，进入后续登录态链路 |

这样分开的原因很直接：

- 初始化管理员是一次性系统引导动作
- 登录是长期运行中的认证入口

::: warning ⚠️ 管理员初始化不是常规用户创建入口
`/api/v1/setup/init` 只适合系统第一次引导时创建首个管理员。后续新增普通账号，应该走用户管理模块，而不是反复调用初始化接口。
:::

## 本节会改什么

当前仓库里，这一节主要对应下面这些代码落点：

```text
server/
├─ internal/
│  ├─ model/
│  │  ├─ user.go
│  │  └─ user_role.go
│  ├─ module/
│  │  ├─ auth/
│  │  │  ├─ dto.go
│  │  │  ├─ repository.go
│  │  │  ├─ login_service.go
│  │  │  ├─ login_handler.go
│  │  │  └─ routes.go
│  │  └─ setup/
│  │     └─ routes.go
│  └─ handler/
│     └─ setup/
│        └─ setup.go
```

| 位置 | 用途 |
| --- | --- |
| `model/user.go` | 定义后台用户账号、密码哈希和状态 |
| `model/user_role.go` | 定义用户与角色关系 |
| `module/auth/login_service.go` | 负责用户名密码校验和登录日志写入 |
| `module/auth/login_handler.go` | 负责登录请求绑定与响应输出 |
| `module/auth/routes.go` | 注册 `/api/v1/auth/login` |
| `module/setup/routes.go` | 注册 `/api/v1/setup/init` |

## 先创建数据表

这一节会用到两张核心表：

- `sys_user`
- `sys_user_role`

它们分别解决：

- 谁是后台用户
- 这个用户拥有哪几个角色

字段和索引详情见：

- [数据库建表语句 - `sys_user`](/reference/database-ddl#sys-user)
- [数据库建表语句 - `sys_user_role`](/reference/database-ddl#sys-user-role)

## 认证起步为什么先要有用户模型

后面的 Token、中间件、菜单和权限，本质上都依赖一个稳定前提：

> 登录成功后，系统必须能唯一确认“当前是谁”。

所以 `sys_user` 至少要先承载这些信息：

- `username`
- `password_hash`
- `nickname`
- `status`
- `department_id`

其中这一节最关键的不是字段多不多，而是两条底线：

- 密码只能存哈希，不能存明文
- 用户状态必须能直接阻断登录

::: warning ⚠️ 这里存的是 `password_hash`，不是密码
当前项目使用 `bcrypt` 处理密码。登录时也不是把明文密码查出来对比，而是用哈希校验函数判断输入是否正确。
:::

## 初始化管理员为什么仍然保留独立入口

企业级后台底座通常都会先有一个“系统第一位管理员”的引导过程。当前主线里，它仍然通过 `/api/v1/setup/init` 完成，原因是：

- 默认角色和菜单由迁移脚本初始化
- 首个管理员账号由使用者自己决定用户名和密码
- 初始化完成后，后续账号创建就应该进入正式用户管理流程

这也意味着当前系统启动时不会直接硬编码一个明文管理员账号，而是把“首个管理员是谁”交给初始化阶段明确落定。

## 登录入口现在已经进入最终版认证模块

这一节最重要的变化，不是“能登录”，而是登录链路已经进入最终结构。  
当前 `/api/v1/auth/login` 不再直接依赖一个全局 `LoginHandler` 自己完成全部逻辑，而是拆成：

- `repository` 查询用户
- `login_service` 校验密码、签发前置数据、写登录日志
- `login_handler` 只负责协议层

这一步很关键，因为后面你再补：

- 登录失败限流
- 首次改密
- 登录设备记录
- 多种登录方式

都不需要再回到一个巨型 Handler 里硬扩。

## 本节先验证什么

这一节建议先验证 3 件事：

### 1. 首个管理员可以被初始化

调用：

```text
POST /api/v1/setup/init
```

应该能创建一名管理员，并绑定到 `super_admin`。

### 2. 登录接口能区分三类结果

当前主线里，`/api/v1/auth/login` 至少要能正确区分：

- 用户不存在或密码错误
- 用户已被禁用
- 用户名密码正确

### 3. 登录成功时已经进入最终版认证模块

这一步先不要求你完全理解 `service / repository` 的每一层细节，但要先建立一个判断：

> 现在的登录接口已经不是“先跑通再说”的临时写法，而是后续会持续扩展的正式模块入口。

## 怎么验证这一节已经做成

### 1. 后端构建通过

在 `server/` 目录执行：

```bash
go test ./...
```

应该能看到 `internal/module/auth` 已经进入编译链路。

### 2. 首个管理员初始化成功

调用：

```text
POST /api/v1/setup/init
```

成功后，数据库里应该能看到：

- `sys_user` 写入首个管理员
- `sys_user_role` 绑定到 `super_admin`

### 3. 登录接口能返回基础登录结果

调用：

```text
POST /api/v1/auth/login
```

用户名和密码正确时，应该至少能返回：

- `user_id`
- `username`
- `nickname`

下一节再继续把它扩成完整登录态。

## 本节最关键的收获

这一节真正建立的判断标准是：

> 认证主线不是从 JWT 开始，而是从“系统先拥有一套稳定的用户身份入口”开始。

用户模型、首个管理员初始化和登录入口一旦收稳，后面的 Token、中间件、菜单和权限，才有了真正可以长期扩展的基础。
