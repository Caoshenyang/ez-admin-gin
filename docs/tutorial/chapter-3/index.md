# 第 3 章：认证与权限

前一章已经把后端基础设施搭好了：配置、日志、数据库、Redis、统一响应和路由入口都已经具备。现在开始补齐后台底座最核心的安全能力：谁可以登录、登录后怎么识别身份、不同角色能访问哪些接口和菜单。

::: tip 本章怎么验证
本章完成后，后端应该能完成登录认证，受保护接口需要携带有效 Token 才能访问，并能根据角色权限控制接口和菜单。
:::

## 开始前准备

进入本章前，建议先确认下面几件事：

- 后端可以正常启动。
- `/health` 和 `/api/v1/system/health` 都能返回成功响应。
- 本地数据库和 Redis 容器处于运行状态。
- 第二章中的统一响应、路由分组和健康检查已经完成。

::: warning ⚠️ 认证功能不要只看“能登录”
本章验证时不能只看登录接口是否返回 Token。还要确认：无 Token 会被拒绝，错误 Token 会被拒绝，权限不足也会被拒绝。
:::

## 本章会完成什么

本章会按下面的顺序推进：

| 小节 | 完成什么 | 验证重点 |
| --- | --- | --- |
| 用户模型与登录 | 创建用户表结构，提供用户名密码登录接口 | 正确账号能登录，错误账号或密码会失败 |
| Token 签发与解析 | 生成和解析登录 Token | Token 能表达用户身份和过期时间 |
| 登录校验中间件 | 保护需要登录的接口 | 不带 Token 不能访问受保护接口 |
| RBAC 角色权限模型 | 建立用户、角色、权限之间的关系 | 一个用户可以拥有角色，角色可以拥有权限 |
| 接口级权限控制 | 用策略判断接口访问权限 | 权限不足时返回统一错误 |
| 角色菜单权限 | 根据权限返回管理台菜单 | 不同角色看到不同菜单 |

::: details 为什么先做认证，再做权限
认证回答“你是谁”，权限回答“你能做什么”。如果还没有稳定的登录身份，就很难判断接口和菜单应该按谁的权限返回。
:::

```text

```
