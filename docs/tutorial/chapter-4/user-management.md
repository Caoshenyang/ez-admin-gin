# 用户管理

这一节不再把用户管理写成“一个 Handler 里把所有事情做完”的示例，而是直接按最终版结构落地。用户模块会成为整套后台底座里的第一块标准答案：它既有 `dto / repository / service / handler / routes / datascope` 的职责边界，也已经开始真正使用数据权限。

::: tip 🎯 本节目标
完成后，`/api/v1/system/users` 这组接口会由独立模块负责，用户列表和编辑类接口会按当前登录人的数据范围过滤，岗位归属会正式进入用户模块和管理台，教程也会第一次完整展示“企业级模块应该怎么长出来”。
:::

## 本节会改什么

这次不再往全局 `internal/handler/system/` 里继续堆代码，而是直接创建用户模块目录：

```text
server/
├─ internal/
│  ├─ module/
│  │  ├─ iam/
│  │  │  └─ user/
│  │  │     ├─ datascope.go
│  │  │     ├─ dto.go
│  │  │     ├─ entity.go
│  │  │     ├─ handler.go
│  │  │     ├─ policy.go
│  │  │     ├─ repository.go
│  │  │     ├─ routes.go
│  │  │     └─ service.go
│  │  └─ system/
│  │     └─ routes.go
│  └─ platform/
│     └─ datascope/
│        └─ datascope.go
```

| 文件 | 作用 |
| --- | --- |
| `dto.go` | 请求参数、响应结构、分页和状态校验 |
| `entity.go` | 当前模块使用到的实体别名 |
| `repository.go` | 用户查询、角色关系读取、岗位关系读取、关系替换、部门和角色/岗位可用性校验 |
| `service.go` | 业务规则、事务边界、密码哈希、状态保护、用户岗位归属收口 |
| `handler.go` | HTTP 协议层绑定与输出 |
| `routes.go` | 注册 `/api/v1/system/users` 相关路由 |
| `datascope.go` | 固定用户资源的数据权限接入方式 |
| `policy.go` | 固定用户模块使用的权限码命名 |

## 为什么用户模块要先这样拆

用户管理表面上只是一个 CRUD 模块，但它天然带着很多后台底座的关键约束：

- 创建用户要校验用户名唯一
- 修改用户要保护当前登录用户，避免误禁用
- 分配角色要校验角色是否存在且启用
- 维护岗位归属要校验岗位是否存在且启用
- 用户列表和详情要按数据权限过滤
- 后续还会继续接真实业务模块

如果这些规则继续堆在一个 Handler 里，后面每加一个约束，查询和事务都会更难收拾。  
所以用户模块很适合作为“最终版结构”的第一块示范。

## 最终版用户模块长什么样

### 1. `handler` 只管协议层

`handler.go` 只做几件事：

- 绑定 Query / JSON 参数
- 从 Gin 上下文里读取当前登录人的 `Actor`
- 调用 Service
- 把 Service 的结果写回响应

它不再直接拼接 SQL，也不在这里控制事务。

### 2. `service` 负责业务规则和事务边界

`service.go` 负责：

- 创建用户时生成密码哈希
- 修改用户时保护当前登录用户
- 更新角色时整体替换关系
- 用事务包住“校验 + 写入 + 关系更新”这类操作

也就是说，Service 是真正决定“业务能不能这样做”的那一层。

### 3. `repository` 负责查询和持久化

`repository.go` 负责：

- 查询当前数据范围内的用户列表
- 读取用户角色关系
- 读取用户岗位关系
- 按数据权限查单个用户
- 校验部门、角色、岗位是否可用
- 创建用户和替换用户角色 / 岗位集合

这样后续如果部门管理、岗位管理、真实业务模块也要接数据权限，就可以沿用同样的职责边界。

### 4. `datascope.go` 固定资源级规则

用户资源既有部门归属，也有“本人”维度，所以模块内单独放了 `datascope.go`，把它的数据权限接法固定下来：

```go
func applyDataScope(db *gorm.DB, actor datascope.Actor) *gorm.DB {
	return db.Scopes(datascope.UserQueryScope(db, actor, "department_id", "id"))
}
```

这样做的关键价值不是“少写一行代码”，而是：  
以后只要看到这个模块的 `datascope.go`，就能知道“用户资源到底按什么字段过滤”。

## 路由怎么挂到系统模块里

系统总路由不再自己 new 一个 `UserHandler`，而是直接把 `/users` 这组接口交给用户模块注册：

```go
system.GET("/health", health.Check)
iamUserModule.RegisterRoutes(system, iamUserModule.RouteOptions{ // [!code ++]
	DB:  opts.DB, // [!code ++]
	Log: opts.Log, // [!code ++]
}) // [!code ++]
system.GET("/roles", roles.List)
```

这一步虽然看起来只是“调用了一个注册函数”，但它背后代表的是一个很重要的方向：

- `system/routes.go` 只负责聚合模块
- 用户模块自己的依赖和路由由自己管理
- 后续角色、部门、岗位、字典、附件中心都可以按同样方式继续拆

## 用户资源的数据权限已经怎么生效

当前主线里，用户模块已经接入了查询级数据权限：

- 用户列表会按当前登录人的 `Actor` 范围过滤
- 用户编辑、状态修改、角色绑定也会先走范围内查询
- 超出当前范围的用户，接口会按“用户不存在”处理

::: warning ⚠️ 这里的数据权限是“查询级过滤”，不是前端自己藏按钮
前端按钮显隐只是体验层，真正的约束必须发生在后端查询和写入链路里。否则只要绕过页面直接调接口，数据权限就形同虚设。
:::

## 这组接口仍然保持什么样的对外形态

虽然内部已经拆成模块，但对外接口形态保持不变：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/system/users` | 用户列表 |
| `POST` | `/api/v1/system/users` | 创建用户 |
| `POST` | `/api/v1/system/users/:id/update` | 编辑用户基础信息 |
| `POST` | `/api/v1/system/users/:id/status` | 修改用户状态 |
| `POST` | `/api/v1/system/users/:id/roles` | 更新用户角色 |

这样做的好处是：  
教程和实现可以继续朝企业级结构演进，但管理台和外部调用方不需要跟着承受不必要的接口震荡。

## 怎么验证这一节已经做成

### 1. 后端构建通过

在 `server/` 目录执行：

```bash
go test ./...
```

应该能看到 `internal/module/iam/user` 已经进入编译链路，并且整体通过。

### 2. 用户接口仍然可访问

登录拿到 Token 后，请求：

```text
GET /api/v1/system/users?page=1&page_size=10
```

应该仍然能拿到分页结果，说明路由已经成功切到新模块。

### 3. 创建和编辑用户时可以维护岗位归属

创建用户请求里现在除了角色，还可以直接带 `post_ids`：

```json
{
  "username": "demo",
  "password": "Demo@123456",
  "nickname": "演示用户",
  "department_id": 1,
  "status": 1,
  "role_ids": [1],
  "post_ids": [1, 3]
}
```

如果岗位不存在或已禁用，后端会直接拒绝这次保存。

### 4. 用户管理页现在会把岗位归属带进最终版表单

管理台里的用户弹窗现在会直接提供岗位多选：

- 新增用户时可以同时选择默认角色和岗位归属
- 编辑用户时会保留已有 `department_id`，避免无意间把部门归属清空
- 岗位归属单独表达“人在哪些岗位上”，不会和角色权限混为一谈

### 5. 列表和编辑开始受数据权限约束

当前登录用户的数据范围如果不是 `all`，那么：

- 用户列表不会返回超出范围的数据
- 编辑、禁用、改角色这些接口也不能越权操作范围外用户

这一步和第 5 章的组织体系与数据权限是同一条主线，不是额外加的小功能。

## 本节最关键的收获

这一节真正要建立的不是“怎么写用户 CRUD”，而是一个判断标准：

> 当一个模块开始同时承担 CRUD、角色/岗位关系写入、权限校验、数据范围过滤和事务边界时，它就应该进入 `repository / service / handler / datascope` 这套最终结构。

这也是为什么用户模块会成为整套教程里第一块标准答案。
