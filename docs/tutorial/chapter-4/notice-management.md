# 公告管理

日志能力补齐之后，第四章再收一块很典型的系统内容模块：公告管理。它不像用户、角色那样属于 IAM，也不像配置、文件那样偏基础设施，但几乎每个后台都会需要一个“轻量内容发布入口”，所以也值得进入最终版结构。

::: tip 🎯 本节目标
完成后，`/api/v1/system/notices` 这组接口会由独立公告模块负责，后台可以稳定地维护公告标题、正文、排序和启用状态，同时保留清晰的 CRUD 边界。
:::

## 先说明边界

这一节的公告管理，解决的是后台里的轻量内容发布：

- 首页提示公告
- 系统通知横幅
- 模块维护说明
- 管理端可见的运营通知

它不解决下面这些更重的场景：

- 面向大量终端用户的内容中心
- 富媒体专题页
- 带复杂审核流的内容发布系统

::: warning ⚠️ 公告模块不是通用 CMS
这一版的公告管理更接近“后台系统自带的轻量内容维护入口”。如果后面要做面向终端用户的内容平台，应该单独抽更完整的内容域，而不是不断把公告表硬扩成 CMS。
:::

## 本节会改什么

公告模块现在也直接按最终结构落地：

```text
server/
├─ internal/
│  ├─ model/
│  │  └─ notice.go
│  └─ module/
│     └─ system/
│        ├─ notice/
│        │  ├─ dto.go
│        │  ├─ entity.go
│        │  ├─ handler.go
│        │  ├─ policy.go
│        │  ├─ repository.go
│        │  ├─ routes.go
│        │  └─ service.go
│        └─ routes.go
```

| 位置 | 用途 |
| --- | --- |
| `dto.go` | 请求体、响应结构、分页边界和公告状态校验 |
| `repository.go` | 公告分页查询、单条读取和持久化 |
| `service.go` | 标题与状态规则、事务边界和返回组装 |
| `handler.go` | HTTP 协议层绑定与输出 |
| `routes.go` | 注册 `/api/v1/system/notices` 相关路由 |
| `policy.go` | 固定公告模块权限码元信息 |

## 先创建数据表

本节复用已经存在的 `sys_notice`，用于保存后台公告内容。

`sys_notice` 表保存后台公告标题、正文、排序和启用状态。字段和索引详情见 [数据库建表语句 - `sys_notice`](/reference/database-ddl#sys-notice)。

## 公告模块现在对外长什么样

当前主线保留 4 个稳定接口：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/system/notices` | 公告分页列表 |
| `POST` | `/api/v1/system/notices` | 创建公告 |
| `POST` | `/api/v1/system/notices/:id/update` | 编辑公告 |
| `POST` | `/api/v1/system/notices/:id/status` | 修改公告状态 |

这组接口刻意保持克制，不急着一口气加：

- 删除
- 富文本附件
- 定时发布
- 阅读统计

因为这一轮的重点是把系统模块的结构边界收稳，而不是把每个内容模块都做成重型产品。

## 为什么公告也值得单独成模块

公告表面上只是一个简单 CRUD，但它同时牵涉：

- 后台内容维护入口
- 仪表盘或首页内容消费入口
- 启用状态和排序规则

如果继续把这些逻辑留在全局 Handler 里，后面一旦要补“首页最新公告”“公告摘要卡片”“按状态筛选”，职责会越来越散。  
所以公告模块现在也进入标准结构：

- `repository` 负责查询和持久化
- `service` 负责标题、状态和事务边界
- `handler` 只负责请求绑定和响应输出
- `policy` 负责固定权限码元信息

## 模块内部收了哪些规则

### 1. 标题和状态先统一在模块内部校验

当前主线保留三条基础规则：

- 标题不能为空
- 标题长度不能超过 128 个字符
- 状态只能是启用或禁用

这些规则现在都收在模块内部，而不是散落在页面和全局 Handler 里。这样后面如果要补摘要、封面或发布时间，扩展位置会很明确。

### 2. 公告模块先聚焦“维护后台内容”

这版公告管理不会直接承担“前台内容分发”职责，但它已经为后续仪表盘和首页消费公告提供了稳定的数据源。也就是说：

- 这一节先把后台维护入口做好
- 后面再由仪表盘或首页按需要读取启用中的公告

### 3. 系统聚合路由只负责装配

`/api/v1/system/notices` 现在不再由旧的全局 `NoticeHandler` 直接挂路由，而是像配置、文件、日志一样，通过系统聚合路由统一装配。

这一步的意义是：

- 公告模块正式进入标准落点
- 第四章的系统模块主线形成一致结构
- 第六章讲模块化接入规范时，公告也能作为完整示例之一

## 怎么验证这一节已经做成

### 1. 后端构建通过

在 `server/` 目录执行：

```bash
go test ./...
```

应该能看到 `internal/module/system/notice` 已经进入编译链路。

### 2. 公告接口已经切到新模块

登录后访问：

```text
GET /api/v1/system/notices?page=1&page_size=10
```

应该仍然能拿到分页结果，说明系统聚合路由已经切到新模块。

### 3. 创建、编辑和状态切换规则仍然生效

当前主线里，公告模块仍然保持这些规则：

- 标题不能为空
- 状态只能在启用和禁用之间切换
- 编辑和状态修改都必须先命中真实公告记录

这说明这次重构只是在收口职责，不是在放松规则。

## 本节最关键的收获

这一节真正建立的判断标准是：

> 后台里的轻量内容模块，虽然没有复杂的组织数据权限，但只要它同时牵涉后台维护入口、内容消费入口和状态排序规则，就值得进入独立模块，而不是继续堆在全局 Handler 里。

公告模块现在的价值，不只是“能发一条公告”，而是第四章里所有高频系统模块，已经基本都进入了统一的最终版结构。
