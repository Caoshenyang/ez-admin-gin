# 文件上传

系统配置完成后，继续补齐后台底座里另一块高频系统能力：文件上传。它看起来只是“收一个 multipart 文件”，但实际同时牵涉上传白名单、磁盘落盘、元数据入库和静态访问路径，所以也值得收进最终版模块结构。

::: tip 🎯 本节目标
完成后，`/api/v1/system/files` 这组接口会由独立文件模块负责；后端会校验文件大小和后缀，把文件保存到本地目录，并在 `sys_file` 中记录元数据。
:::

## 本节会改什么

文件模块现在按最终结构落地：

```text
server/
├─ internal/
│  ├─ module/
│  │  └─ system/
│  │     └─ file/
│  │        ├─ dto.go
│  │        ├─ entity.go
│  │        ├─ handler.go
│  │        ├─ policy.go
│  │        ├─ repository.go
│  │        ├─ routes.go
│  │        └─ service.go
│  ├─ model/
│  │  └─ system_file.go
│  └─ module/
│     └─ system/
│        └─ routes.go
```

| 位置 | 用途 |
| --- | --- |
| `dto.go` | 分页查询结构、上传返回结构和后缀白名单辅助函数 |
| `repository.go` | 文件分页查询与文件记录持久化 |
| `service.go` | 上传校验、落盘、SHA256 计算和事务边界 |
| `handler.go` | HTTP 协议层绑定与输出 |
| `routes.go` | 注册 `/api/v1/system/files` 相关路由 |
| `policy.go` | 固定文件模块权限码元信息 |

## 先创建数据表

本节新增 `sys_file`，用于保存文件上传后的元数据，文件内容仍然保存在本地上传目录中。

`sys_file` 表保存文件上传后的元数据，文件内容保存在本地上传目录。字段和索引详情见 [数据库建表语句 - `sys_file`](/reference/database-ddl#sys-file)。

## 文件如何保存

本节采用下面的保存规则：

```text
server/uploads/
└─ 20260423/
   └─ 20260423153000_a1b2c3d4e5f6a7b8.txt
```

| 数据 | 示例 | 说明 |
| --- | --- | --- |
| 原始文件名 | `测试文件.txt` | 用户上传时的文件名 |
| 保存文件名 | `20260423153000_a1b2c3d4e5f6a7b8.txt` | 后端生成，避免重名 |
| 磁盘路径 | `uploads/20260423/...txt` | 写入 `sys_file.path` |
| 访问地址 | `/uploads/20260423/...txt` | 写入 `sys_file.url` |

::: warning ⚠️ 不要直接使用用户上传的文件名作为保存文件名
用户上传的文件名可能重复，也可能包含不适合直接拼接路径的字符。后端应该保留原始文件名用于展示，同时生成新的安全文件名用于保存。
:::

## 接口规划

本节实现 2 个接口：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/system/files` | 文件分页列表 |
| `POST` | `/api/v1/system/files` | 上传文件 |

上传成功后返回文件记录，其中 `url` 可以直接访问：

```json
{
  "id": 1,
  "original_name": "test-upload.txt",
  "file_name": "20260423153000_a1b2c3d4e5f6a7b8.txt",
  "url": "/uploads/20260423/20260423153000_a1b2c3d4e5f6a7b8.txt"
}
```

## 为什么文件上传也值得单独成模块

文件上传并不只是一个“收表单然后写数据库”的接口，它至少同时承担了四件事：

- 上传白名单和大小限制校验
- 磁盘目录创建和文件落盘
- SHA256 计算与元数据入库
- 对外可访问 URL 的统一生成

如果继续把这些逻辑都留在一个全局 Handler 里，后面一旦再补“云存储适配”“附件分类”“清理策略”，职责会越来越难收。  
所以文件模块现在也进入标准结构：

- `repository` 负责查询和记录持久化
- `service` 负责上传规则和落盘流程
- `handler` 只负责协议层
- `policy` 固定权限码，不再散落命名

## 文件模块内部收了哪些规则

### 1. 上传白名单和大小限制先由后端守住

当前主线里，文件模块会统一校验：

- 文件不能是空文件
- 大小不能超过 `upload.max_size_mb`
- 后缀必须命中 `upload.allowed_exts`

::: warning ⚠️ 后缀白名单只是第一层限制
这一节先用后缀白名单和大小限制完成基础能力。真实项目里如果允许上传可执行风险更高的文件，还应该结合 MIME 检测、病毒扫描、私有访问策略等继续加强。
:::

## 模块切到系统聚合路由后意味着什么

`/api/v1/system/files` 现在也不再由旧的全局 `FileHandler` 直接挂在系统路由上，而是像菜单和配置一样，通过系统聚合路由统一装配。

这一步的意义很直接：

- 上传配置和文件模块依赖在一个地方收口
- 文件模块自己管理路由和服务依赖
- 后续如果要加“附件中心”“存储策略切换”“文件清理任务”，可以在模块内部继续长

## 🛠️ 初始化文件上传权限和菜单

文件上传的权限和菜单已经在数据库迁移文件中初始化。迁移文件会在服务启动时自动执行，创建文件上传相关的权限策略和菜单数据。

::: tip 💡 权限和菜单初始化
- 权限策略：在 `migrations/{postgres,mysql}/000002_seed_data.up.sql` 中插入文件上传接口的 Casbin 规则
- 菜单数据：在同一迁移文件中插入文件上传菜单和按钮
- 角色菜单绑定：在同一迁移文件中绑定 `super_admin` 角色到文件上传菜单
:::

## 怎么验证这一节已经做成

### 1. 后端构建通过

```bash
go test ./...
```

应该能看到 `internal/module/system/file` 已经进入编译链路。

### 2. 文件接口已经切到新模块

登录后访问：

```text
GET /api/v1/system/files?page=1&page_size=10
```

应该仍然能拿到分页结果，说明系统聚合路由已经切到新模块。

### 3. 上传规则和落盘规则仍然生效

当前主线里，文件模块仍然保持这些规则：

- 文件名由后端重新生成
- 落盘目录按日期分层
- 元数据写入 `sys_file`
- URL 继续走 `/uploads/...` 公开路径

### 4. 静态访问路径仍然是系统能力的一部分

文件模块虽然只接管 `/api/v1/system/files`，但上传后的资源访问仍然依赖全局静态路由。也就是说，文件模块负责“上传和记录”，静态文件映射继续负责“怎么被访问”。

## 本节最关键的收获

这一节真正建立的判断标准是：

> 只要一个系统能力同时牵涉“协议层上传 + 磁盘落盘 + 元数据入库 + 对外访问路径”，它就应该进入独立模块，而不是继续停留在全局 Handler 里。

文件模块现在的价值，不只是“能传一个文件”，而是它已经成为后续附件中心、存储策略扩展和文件治理能力的稳定起点。
