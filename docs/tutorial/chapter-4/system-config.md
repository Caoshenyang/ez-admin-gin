# 系统配置

前面已经把用户、角色和菜单这些核心入口收进最终结构了。系统配置是下一块很典型的“系统底座能力”：它不直接属于 IAM，但几乎每个后台项目都会依赖它，所以也值得尽早进入稳定的模块边界。

::: tip 🎯 本节目标
完成后，`/api/v1/system/configs` 这组接口会由独立的系统配置模块负责，后台可以维护配置项，业务代码可以按配置键读取值，并优先命中 Redis 缓存。
:::

## 先说明边界

这一节的系统配置，适合放这类内容：

- 站点标题
- 默认上传目录
- 首页公告开关
- 某个模块的默认分页大小

不适合放这类内容：

- 数据库密码
- JWT 密钥
- 第三方平台 Access Key / Secret
- Redis 连接信息

::: warning ⚠️ 系统配置不是密钥管理
需要跟随部署环境变化、且具备敏感性的内容，仍然应该放在 `.env` 或配置文件中，由运维配置和环境变量管理。系统配置更适合放“业务可调参数”。
:::

## 本节会改什么

系统配置现在也直接按最终结构落地：

```text
server/
├─ internal/
│  ├─ module/
│  │  └─ system/
│  │     └─ config/
│  │        ├─ dto.go
│  │        ├─ entity.go
│  │        ├─ handler.go
│  │        ├─ policy.go
│  │        ├─ repository.go
│  │        ├─ routes.go
│  │        └─ service.go
│  ├─ model/
│  │  └─ system_config.go
│  └─ module/
│     └─ system/
│        └─ routes.go
```

| 位置 | 用途 |
| --- | --- |
| `dto.go` | 请求体、响应结构、分页与配置编码校验 |
| `repository.go` | 配置分页查询、按 key 读取、唯一性校验和持久化 |
| `service.go` | 事务边界、缓存同步和读取策略 |
| `handler.go` | HTTP 协议层绑定与输出 |
| `routes.go` | 注册 `/api/v1/system/configs` 相关路由 |
| `policy.go` | 固定系统配置模块的权限码元信息 |

## 先创建数据表

本节新增 `sys_config`，用于保存后台可维护的普通业务配置。

`sys_config` 表保存后台可维护的普通业务配置，按配置键存储。字段和索引详情见 [数据库建表语句 - `sys_config`](/reference/database-ddl#sys-config)。

## 接口规划

本节实现 5 个接口：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/system/configs` | 配置分页列表 |
| `POST` | `/api/v1/system/configs` | 创建配置 |
| `POST` | `/api/v1/system/configs/:id/update` | 编辑配置 |
| `POST` | `/api/v1/system/configs/:id/status` | 修改配置状态 |
| `GET` | `/api/v1/system/configs/value/:key` | 按配置键读取启用中的配置值 |

其中 `/api/v1/system/configs/value/:key` 有两个作用：

1. 给后续业务模块提供统一读取入口。
2. 方便这一节验证 Redis 缓存是否生效。

## 为什么系统配置也值得单独成模块

系统配置表面上像“一个普通键值表”，但它其实同时承担了三件事：

- 提供后台配置管理入口
- 给业务代码暴露统一读取方式
- 负责数据库与 Redis 缓存之间的同步

如果继续把这些逻辑堆在一个全局 Handler 里，后面一旦再加“按分组导出”“配置刷新”“模块默认值回填”，职责会很快变得发散。  
所以系统配置现在也进入标准结构：

- `repository` 负责查询和持久化
- `service` 负责事务和缓存同步
- `handler` 只做请求绑定与响应输出
- `policy` 固定权限码命名，不再散落在别处

## 系统配置模块内部收了哪些规则

### 1. 配置键创建后保持稳定

创建接口会接收 `key`，但编辑接口不会再允许修改它。原因和菜单编码类似：

- 业务代码会按 key 读取配置
- Redis 缓存键直接依赖 key
- 文档和默认配置约定也会引用 key

::: warning ⚠️ 配置值统一按字符串存储
这一版不在配置表里区分整数、布尔值或 JSON。后续业务模块如果需要布尔值或数字，读取配置后再自行解析。
:::

::: details 为什么缓存异常不让接口直接失败
系统配置的缓存是性能优化，不是唯一数据源。即使 Redis 临时不可用，也应该还能回退到数据库读取；否则一个缓存故障会把整个后台管理能力一起拖下去。
:::

## 模块切到系统聚合路由后意味着什么

`/api/v1/system/configs` 现在不再由旧的全局 `SystemConfigHandler` 直接挂在路由上，而是像其他最终版模块一样，通过系统聚合路由统一注册。

这一步最重要的意义是：

- `module/system/routes.go` 只负责装配
- 配置模块自己的依赖和路由由自己管理
- 后续文件、日志、公告这些系统能力都可以沿着同一条路径继续收口

## 🛠️ 初始化系统配置权限和菜单

系统配置的权限和菜单已经在数据库迁移文件中初始化。迁移文件会在服务启动时自动执行，创建系统配置相关的权限策略和菜单数据。

::: tip 💡 权限和菜单初始化
- 权限策略：在 `migrations/{postgres,mysql}/000002_seed_data.up.sql` 中插入系统配置接口的 Casbin 规则
- 菜单数据：在同一迁移文件中插入系统配置菜单和按钮
- 角色菜单绑定：在同一迁移文件中绑定 `super_admin` 角色到系统配置菜单
:::

::: details 为什么配置键创建后不允许修改
配置键会被业务代码、缓存键和接口路径共同依赖。创建后保持稳定，后续排查问题会轻松很多；如果真的要改，一般会单独做迁移或改名方案，而不是在后台里随手改。
:::

## 怎么验证这一节已经做成

### 1. 后端构建通过

```bash
go test ./...
```

应该能看到 `internal/module/system/config` 已经进入编译链路。

### 2. 配置接口已经切到新模块

登录后访问：

```text
GET /api/v1/system/configs?page=1&page_size=10
```

应该仍然能拿到分页结果，说明系统聚合路由已经切到新模块。

### 3. 配置键和缓存规则仍然生效

当前主线里，系统配置仍然保持这些规则：

- 配置键创建后保持稳定
- 启用中的配置优先命中 Redis
- Redis 异常时回退到数据库读取
- 禁用配置时同步清理缓存

### 4. 系统配置仍然不是密钥管理

这块边界仍然不能模糊：JWT 密钥、数据库密码、第三方 Access Key 这些内容，依旧应该留在环境变量和部署配置里。

## 本节最关键的收获

这一节真正建立的判断标准是：

> 只要一个系统能力同时牵涉“后台管理入口 + 业务侧读取入口 + 缓存同步”，它就值得进入独立模块，而不是继续塞在全局 Handler 里。

系统配置模块现在的价值，不只是“能维护几条键值对”，而是它已经成为后续各类业务模块复用运行期配置的稳定底座。
