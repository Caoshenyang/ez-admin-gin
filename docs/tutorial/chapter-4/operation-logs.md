# 操作日志

文件上传之后，继续补齐企业级后台里另一块很典型的审计能力：操作日志。它的价值不是“多一个列表页”，而是把后台写操作沉淀成稳定的审计记录，让后续排障、风控和责任追踪有据可查。

::: tip 🎯 本节目标
完成后，后台的写操作仍然会由中间件自动写入 `sys_operation_log`，而 `/api/v1/system/operation-logs` 这组查询接口会切到独立模块，由标准的 `dto / repository / service / handler / routes / policy` 结构负责对外提供列表能力。
:::

## 先说明边界

操作日志在最终版结构里分成两部分：

| 部分 | 职责 |
| --- | --- |
| `middleware/operation_log.go` | 在请求结束后自动写入审计记录 |
| `module/system/operationlog` | 提供后台查询入口 |

这两个部分故意不混在一起。

::: warning ⚠️ 不要把“写日志”和“查日志”塞进同一个 Handler
操作日志的写入属于横切能力，应该跟着请求链路自动发生；查询操作日志才是一个后台模块。如果把两者揉成一处，后面一旦再补保留策略、归档或链路追踪字段，职责会很快混乱。
:::

## 本节会改什么

操作日志查询入口现在直接按最终结构落地：

```text
server/
├─ internal/
│  ├─ middleware/
│  │  └─ operation_log.go
│  ├─ model/
│  │  └─ operation_log.go
│  └─ module/
│     └─ system/
│        ├─ operationlog/
│        │  ├─ dto.go
│        │  ├─ entity.go
│        │  ├─ handler.go
│        │  ├─ policy.go
│        │  ├─ repository.go
│        │  ├─ routes.go
│        │  └─ service.go
│        └─ routes.go
```

| 位置 | 用途 |
| --- | --- |
| `middleware/operation_log.go` | 请求结束后自动写审计记录 |
| `dto.go` | 分页查询结构、成功状态过滤器和返回结构 |
| `repository.go` | 操作日志分页查询和过滤条件收口 |
| `service.go` | 分页边界、成功状态参数校验和返回组装 |
| `handler.go` | HTTP 协议层绑定与输出 |
| `routes.go` | 注册 `/api/v1/system/operation-logs` 路由 |
| `policy.go` | 固定操作日志模块权限码元信息 |

## 先创建数据表

本节复用已经存在的 `sys_operation_log`，用于保存后台用户的关键写操作审计记录。

`sys_operation_log` 表保存后台用户的写操作审计记录，不做逻辑删除。字段和索引详情见 [数据库建表语句 - `sys_operation_log`](/reference/database-ddl#sys-operation-log)。

## 为什么操作日志要拆成“中间件 + 模块”

操作日志和用户、角色、菜单这类普通后台资源不一样，它天然有两种完全不同的职责：

- 请求链路结束后自动记一条日志
- 管理员按用户名、方法、路径和成功状态查询日志

其中第一件事适合放在中间件里，因为它依赖：

- 当前用户是谁
- 当前接口最终状态码是多少
- 请求耗时多久
- 请求是成功还是失败

这些信息都只有在 `c.Next()` 之后才完整。  
第二件事才属于标准后台模块，所以现在它被收进 `module/system/operationlog`。

## 操作日志模块现在对外长什么样

当前主线保留 1 个稳定查询接口：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/system/operation-logs` | 操作日志分页列表 |

支持的查询条件继续保持克制：

- `page`
- `page_size`
- `username`
- `method`
- `path`
- `success`

::: details 为什么暂时不做更复杂的筛选
这一版先把最终结构和查询边界收稳，不急着一口气补时间区间、模块名、请求链路 ID、批量导出等更重的能力。等后面把登录日志、公告等系统模块也收口后，再继续往“更完整的审计中心”扩展会更自然。
:::

## 模块内部收了哪些规则

### 1. 成功状态过滤器先统一在 `dto.go` 收口

`success` 现在仍然支持：

- `true`
- `false`
- `1`
- `0`

但解析规则不再散落在 Handler 里，而是由模块内部统一转换。这样后面如果要把查询参数扩成更明确的枚举，改动范围会很小。

### 2. 操作日志模块保持只读

这一节的操作日志模块只有查询接口，不提供：

- 手工新增
- 手工编辑
- 手工删除

因为操作日志的可信度来自“自动写入、尽量不可篡改”。如果后台可以随意编辑日志，它作为审计依据的价值会迅速下降。

::: warning ⚠️ 操作日志默认不记录请求体
这一版仍然只记录方法、路径、查询参数、用户、IP、状态码和耗时，不原样保存请求体。这样可以先避开密码、Token、文件内容和其他敏感数据直接落表的问题。
:::

### 3. 系统聚合路由只负责装配

`/api/v1/system/operation-logs` 现在不再由旧的全局 `OperationLogHandler` 直接挂路由，而是像菜单、配置、文件一样，通过系统聚合路由统一装配。

这一步的意义是：

- 审计查询能力也进入标准模块落点
- `module/system/routes.go` 只负责装配，不继续扩大查询细节
- 后面补登录日志或审计保留策略时，可以沿着同一模式继续长

## 操作日志中间件还保留了什么规则

查询入口虽然进了最终结构，但“怎么写日志”这条链路没有变形，当前主线仍然保留这几个判断：

- 只记录后台写操作
- 只在请求结束后写日志
- 已登录但权限失败的写操作，也应该留下失败记录
- 静态资源和未匹配到路由的请求不作为后台操作记录

这也是为什么操作日志中间件要放在认证之后、权限中间件之前。

## 怎么验证这一节已经做成

### 1. 后端构建通过

在 `server/` 目录执行：

```bash
go test ./...
```

应该能看到 `internal/module/system/operationlog` 已经进入编译链路。

### 2. 查询接口已经切到新模块

登录后访问：

```text
GET /api/v1/system/operation-logs?page=1&page_size=10
```

应该仍然能拿到分页结果，说明系统聚合路由已经切到新模块。

### 3. 审计写入能力仍然生效

继续调用一个后台写操作，例如：

- 上传文件
- 修改用户状态
- 调整角色菜单授权

执行后再查操作日志列表，应该能看到对应记录。说明这次重构只是在收口查询入口，没有破坏中间件写入链路。

## 本节最关键的收获

这一节真正建立的判断标准是：

> 审计能力不等于“做一个日志列表页”。只要一个能力同时牵涉请求链路自动写入和后台查询入口，就应该拆成“横切中间件 + 独立模块”两部分，而不是继续堆在一个全局 Handler 里。

操作日志现在的价值，不只是“能查最近谁点了什么”，而是它已经开始具备企业级后台审计能力该有的结构边界。
