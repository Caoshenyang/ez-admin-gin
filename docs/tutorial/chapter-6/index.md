# 第 6 章：业务模块接入规范

前面已经完成了后台底座的两条主线：后端有认证、权限和系统模块，前端有登录、布局、菜单和基础页面。现在开始进入更接近真实项目的部分：把一个业务模块接进来，并把接入过程沉淀成可重复使用的规范。

::: tip 🎯 本章怎么验证
本章完成后，你应该能按照固定结构新增一个业务模块，让它同时具备后端接口、权限菜单、数据库结构和前端页面入口。以后再做新的个人项目时，不需要重新摸索模块该放哪里、权限该怎么挂、页面该怎么接。
:::

## 为什么需要这一章

一个后台底座是否真正可复用，不只看它有没有登录、用户、角色和菜单。更关键的是：当你要接入第一个真实业务时，项目结构会不会开始变乱。

常见问题通常出现在这些地方：

- 新表、新模型、新接口分散在不同目录，后来很难回查。
- 权限编码、菜单路由和前端页面命名没有统一规则。
- 接口能跑通，但菜单不可见，或者菜单能点开但接口被权限拦住。
- 示例模块做完一次，下一个模块仍然要靠复制粘贴和猜路径。

这一章要解决的不是“某一个页面怎么写”，而是给业务模块接入定一套稳定路径。

::: details Go vs Java：为什么这里要先定目录约定
在 Java 后台项目里，业务模块常见做法是按包组织 `controller / service / repository / entity`，再通过注解、扫描和配置把它们挂进应用。

当前项目没有引入很重的模块框架，所以更依赖清晰的目录、命名和注册约定。好处是轻，坏处是如果没有规范，新增模块很容易散。第 6 章会把这些“靠经验记住”的规则写成可以照着做的流程。
:::

## 开始前准备

进入本章前，建议先确认下面几件事：

- 后端服务可以正常启动。
- 前端管理台可以登录并进入后台布局。
- 菜单权限已经能控制侧边栏展示。
- `super_admin` 角色可以访问系统管理相关接口。
- 第 4 章和第 5 章中的系统模块接口、页面入口已经基本可用。

::: warning ⚠️ 业务模块不要只验证接口
这一章每完成一个模块接入动作，都要同时检查四个位置：数据库有没有表和初始化数据，后端路由有没有注册，权限和菜单有没有生效，前端页面能不能从侧边栏进入。
:::

## 本章的模块约定

为了让后续小节更容易跟做，本章会默认采用下面这组约定：

- 后端业务模块按领域命名，避免把所有代码继续塞进 `system`。
- 接口路径统一放在 `/api/v1` 下，并保持资源名清晰。
- 权限编码与接口语义对应，便于在角色授权时快速识别。
- 菜单路由与前端页面路径对应，避免出现“菜单名称能看懂，路由却猜不到”的情况。
- 前端页面优先复用第 5 章已经建立的布局、接口请求和类型组织方式。

::: info 本章不是要做复杂插件系统
这里的目标是让个人项目里的业务模块接入更稳定，而不是引入动态插件加载、微前端或代码生成平台。先把手工接入路径变清楚，后面才有资格考虑自动化。
:::
