# 岗位管理与用户归属

部门树稳定下来后，组织体系里还缺最后一块常用主数据：岗位。岗位不像部门那样直接决定树形层级，但它会直接影响用户归属、角色协作和后续通讯录、审批、消息通知这类功能的扩展空间。

::: tip 🎯 本节目标
完成后，岗位模块会像用户、角色、部门一样进入最终版结构，`sys_user_post` 也会正式进入用户模块和管理台，组织闭环不再只停留在模型层。
:::

## 本节会改什么

这次新增的是 `iam/post` 模块：

```text
server/
├─ internal/
│  ├─ module/
│  │  ├─ iam/
│  │  │  └─ post/
│  │  │     ├─ datascope.go
│  │  │     ├─ dto.go
│  │  │     ├─ entity.go
│  │  │     ├─ handler.go
│  │  │     ├─ policy.go
│  │  │     ├─ repository.go
│  │  │     ├─ routes.go
│  │  │     └─ service.go
│  │  └─ system/
│  │     └─ routes.go
│  └─ model/
│     └─ user_post.go
```

| 文件 | 作用 |
| --- | --- |
| `dto.go` | 岗位请求体、响应结构和状态校验 |
| `repository.go` | 岗位查询、编码唯一校验和持久化 |
| `service.go` | 事务边界和岗位规则收口 |
| `handler.go` | HTTP 协议层绑定与输出 |
| `routes.go` | 注册 `/api/v1/system/posts` 相关路由 |
| `datascope.go` | 当前阶段显式声明岗位暂不按部门范围裁剪 |

## 为什么岗位模块现在先独立出来

岗位资源和部门资源的性质不完全一样：

- 部门是组织树的一部分，会直接影响数据权限
- 岗位更像组织标签，重点在“用户挂了哪些岗位”

所以这一轮的策略是：

- 先把岗位模块本身收进最终结构
- 先让岗位成为可管理资源
- `sys_user_post` 作为用户归属关系表先打底
- 再把岗位多选接进用户模块接口和用户管理页

这样做的好处是节奏更稳：岗位模块先稳定成独立资源，再把“用户挂哪些岗位”收口到用户服务里，职责不会混乱。

## 岗位模块现在对外长什么样

当前主线先给岗位模块补 4 个接口：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/system/posts` | 查询岗位列表 |
| `POST` | `/api/v1/system/posts` | 创建岗位 |
| `POST` | `/api/v1/system/posts/:id/update` | 编辑岗位 |
| `POST` | `/api/v1/system/posts/:id/status` | 修改岗位状态 |

它和用户、角色、部门一样，已经通过系统聚合路由挂进 `/api/v1/system` 分组。

## 为什么岗位模块当前不做部门范围裁剪

这轮实现里，岗位模块虽然也放了 `datascope.go`，但当前规则是显式放开：

```go
func applyDataScope(db *gorm.DB) *gorm.DB {
	return db
}
```

这不是偷懒，而是当前主线里的一个明确取舍：

- 岗位本身暂时不直接归属于部门节点
- 岗位更像系统级组织字典
- 等后续真的引入“部门岗位映射”或更复杂的人事结构时，再把岗位资源本身收紧范围

这里故意把这个决定写进模块，而不是省略掉，是为了让后续扩展时知道“现在的范围规则就是有意放开的”。

## `sys_user_post` 现在的角色是什么

当前主线已经有：

- `sys_user.department_id`
- `sys_user_post`

这两者一起表达的是：

- 用户的主归属部门
- 用户挂了哪些岗位

也就是说，组织体系现在已经具备了“部门 + 岗位 + 用户归属”的基础闭环，而且岗位关系已经正式挂进用户模块接口和用户管理页。

这样安排的好处是：

- 组织模型已经定稳
- 关系表已经可用
- 岗位多选进入用户编辑页时，不需要再改底层表设计
- 用户服务已经统一负责岗位可用性校验和关系替换

## 这一节最容易误解的点

最容易出现的误解是：

> 既然有 `sys_user_post`，为什么还要单独讲“岗位管理与用户归属”？

答案是：岗位资源和用户岗位关系是两层不同职责，最好分开理解：

- 岗位模块负责“系统里有哪些岗位”
- 用户模块负责“一个用户挂了哪些岗位”
- `sys_user_post` 只是二者之间的关系表，不应该把岗位资源本身塞回用户模块里

所以这一轮真正完成的是两件事：

1. 组织模型闭环已经成立
2. 用户岗位关系已经通过用户服务、用户接口和用户管理页收口成真实能力

## 怎么验证这一节已经做成

### 1. 后端构建通过

在 `server/` 目录执行：

```bash
go test ./...
```

应该能看到 `internal/module/iam/post` 已经进入编译链路，并整体通过。

### 2. 岗位接口已经进入系统路由

登录后请求：

```text
GET /api/v1/system/posts
```

应该能拿到岗位列表结果。

### 3. 岗位编码仍然保持唯一

创建岗位时，如果重复使用相同 `code`，后端应该直接拒绝。

### 4. 用户管理页已经能维护岗位归属

打开用户管理弹窗后，现在可以直接看到岗位多选：

- 新增用户时可以一起绑定初始岗位
- 编辑用户时可以调整岗位归属
- 列表会把岗位作为独立列展示出来

这说明 `sys_user_post` 已经从“底层关系表”变成“用户模块对外能力”的一部分。

### 5. 组织闭环已经具备三块主数据

到这一步，当前底座已经同时具备：

- 部门：`sys_department`
- 岗位：`sys_post`
- 用户岗位关系：`sys_user_post`

这意味着当前主线已经不需要再为“用户岗位关系”补一轮过渡实现。

## 本节最关键的收获

这一节真正要建立的判断标准是：

> 一个资源即使当前还没有复杂权限规则，只要它会成为长期组织模型的一部分，也应该尽早按最终结构落地，而不是先塞进旧目录凑合用。

岗位模块现在的价值，不只是补了一套 CRUD，而是把组织体系最后一块常用主数据也拉进了统一结构，并且顺利把岗位关系收口进了用户模块。

## 下一步

到这里，组织体系的基础闭环已经成立。下一步最值得做的是继续把菜单、配置、日志这些系统模块也收进最终结构，并让第 5 章把组织与数据权限的正文彻底写透。
