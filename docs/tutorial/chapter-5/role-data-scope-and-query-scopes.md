# 角色数据范围与查询作用域

上一节先把组织模型定稳了，这一节继续往前走：把角色数据范围、自定义部门授权、`Actor` 上下文和查询作用域真正串起来。到了这里，数据权限才算从“表结构设计”进入“可执行规则”。

::: tip 🎯 本节目标
完成后，你应该能清楚回答三件事：角色的数据范围存在哪里、当前登录人怎么拿到这份范围、列表查询为什么应该通过 `gorm.Scopes(...)` 自动过滤。
:::

## 数据权限真正依赖的三层结构

企业级后台里的数据权限，通常至少依赖下面三层：

1. **角色层**：每个角色定义自己的 `data_scope`
2. **登录态层**：请求进入系统后，把用户的角色范围加载到 `Actor`
3. **查询层**：Repository 在查询时通过 `gorm.Scopes(...)` 应用过滤

这三层缺一不可。

如果只有角色层，没有登录态装载，那么每个接口都得重新查一遍角色。  
如果只有登录态层，没有查询层，那么数据范围只存在于内存里，最终不会真正限制 SQL 结果。  
如果只有查询层，但角色和组织关系没定稳，过滤条件就会越来越乱。

## 角色数据范围放在哪里

当前主线里，角色数据范围用两部分表达：

| 结构 | 用途 |
| --- | --- |
| `sys_role.data_scope` | 角色的基础数据范围 |
| `sys_role_data_scope` | 当范围为 `custom_dept` 时，角色被额外授予的部门集合 |

也就是说，角色模块现在不仅能保存名称和状态，还能真正保存“这类角色允许看哪些数据”。

## `Actor` 为什么是关键桥梁

请求通过认证后，系统会构建一个 `Actor`，它代表“当前登录人已经被解析好的权限上下文”。

当前 `Actor` 至少包含：

```go
type Actor struct {
	UserID       uint
	Username     string
	DepartmentID uint
	RoleCodes    []string
	Grants       []Grant
	IsSuperAdmin bool
}
```

这里最重要的是 `Grants`。它会把当前用户所有角色授予的数据范围合并进来，后续查询不需要再重新理解角色表，只需要面向 `Actor` 工作。

## 为什么查询层要用 `gorm.Scopes(...)`

在后台项目里，数据权限最容易失控的地方，就是大家在 Handler 里各写各的过滤条件。

例如：

- 用户列表里手写一次部门过滤
- 用户编辑接口里再手写一次
- 真实业务列表里又手写第三次

时间一长，规则就会开始漂移。  
同一个角色，在列表和详情里看到的范围甚至可能不一样。

所以这条主线里，数据权限统一通过 `gorm.Scopes(...)` 往查询层收：

```go
func applyDataScope(db *gorm.DB, actor datascope.Actor) *gorm.DB {
	return db.Scopes(datascope.UserQueryScope(db, actor, "department_id", "id"))
}
```

这样做的价值不是“看起来更 Go”，而是能把数据权限真正变成模块内部一条稳定规则。

## 多角色并集为什么放在平台层

当前主线固定采用并集规则：

- 只要任一角色是 `all`，结果就是全部数据
- `dept` 和 `custom_dept` 可以并在一起
- `super_admin` 直接绕过数据权限

这套规则现在固化在：

- `/server/internal/platform/datascope/datascope.go`

也就是说，多角色合并不由业务模块自己决定。  
业务模块只负责声明“这个资源应该按什么字段过滤”，不负责重新定义并集语义。

这一步非常重要，因为如果每个模块都可以自己解释多角色合并规则，后面就几乎不可能排查权限问题。

## 用户资源现在是怎么接入的

当前主线里，用户资源已经开始接入这套查询作用域。

它的规则是：

- 部门归属字段：`department_id`
- 本人归属字段：`id`

所以用户模块会这样声明自己的过滤逻辑：

```go
func applyDataScope(db *gorm.DB, actor datascope.Actor) *gorm.DB {
	return db.Scopes(datascope.UserQueryScope(db, actor, "department_id", "id"))
}
```

接下来无论是：

- 列表
- 编辑
- 状态修改
- 角色绑定

只要走的是“先查用户，再执行后续动作”，就能自动复用同一套范围规则。

## 这和 Java 里的常见做法怎么对照

如果你主要做 Java，这里可以这样理解：

- `Actor` 很像登录后的 `UserContext`
- `datascope.Merge(...)` 很像把多个角色权限压成一个 `DataPermissionContext`
- `gorm.Scopes(...)` 很像在 Repository / Mapper 层统一追加查询条件

最大的区别不在思路，而在落点：

- Java 项目里这类能力常常通过 AOP、注解、拦截器或 SQL 片段增强实现
- 当前这套 Go 单体底座更强调“显式调用作用域”，避免规则藏得太深

这也是为什么这里会更偏向“模块内声明 + Repository 复用”，而不是一开始就做成非常魔法化的全局自动注入。

## 怎么验证这一节讲的不是空规则

### 1. `/auth/me` 已经能返回数据范围摘要

登录后访问：

```text
GET /api/v1/auth/me
```

应该能看到：

- `role_codes`
- `is_super_admin`
- `data_scope`

这说明 `Actor` 已经能把当前登录人的数据范围整理出来。

### 2. 角色接口已经支持 `data_scope`

访问角色列表接口时，响应里应该已经能看到：

- `data_scope`
- `custom_department_ids`

这说明角色数据范围已经进入配置层。

### 3. 用户模块已经复用统一查询作用域

当前用户模块的列表和编辑类接口都已经不再自己散写过滤条件，而是通过模块内的 `datascope.go` 统一接到平台层。

## 本节最关键的收获

这一节真正建立的判断标准是：

> 数据权限不是“在 SQL 里多写个 where”这么简单，它是一条从角色配置、到登录态装载、再到查询执行的完整链路。

只有把这条链路一次讲清楚，后面的部门模块、岗位模块和真实业务模块才能真正复用。
