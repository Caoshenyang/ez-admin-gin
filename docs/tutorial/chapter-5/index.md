# 第 5 章：前端管理台

前面已经补齐了后端的认证、权限和系统模块。现在开始搭建前端管理台，让这些接口有一个可视化操作入口。

::: tip 本章怎么验证
本章完成后，管理台应该可以登录、展示后台布局、按后端菜单生成侧边栏，并提供用户、角色、菜单、配置、文件和日志等基础页面入口。
:::

## 开始前准备

进入本章前，建议先确认下面几件事：

- `server` 可以正常启动。
- `admin` 项目已经通过 Vue 官方脚手架创建。
- `admin` 目录下可以执行 `pnpm install`、`pnpm dev`、`pnpm build`。
- `admin` 当前还保留 Vue 脚手架默认示例页面。
- 后端登录接口 `/api/v1/auth/login` 可以正常返回 Token。

::: warning ⚠️ 前端章节不要只看页面能不能打开
这一章会逐步从静态页面走向真实接口。验证时除了看页面显示，还要确认 Token、菜单、接口请求和路由跳转是否符合预期。
:::

## 本章前端约束

从这一章开始，前端页面统一采用 `Naive UI + Tailwind CSS 4` 的混合方案。后面写登录页、后台布局、空状态页和系统页面时，都尽量按下面这组约束保持一致：

- Naive UI 优先负责成熟组件和后台基础结构，包括 `NLayout`、`NMenu`、`NForm`、`NButton`、`NCard` 等。
- Tailwind CSS 4 主要负责页面尺寸约束、栅格补充、间距、背景、颜色微调和响应式布局。
- 页面级视觉可以用 Tailwind 4 组织，但只要 Naive UI 已经提供成熟组件，就优先复用组件能力。
- 后台主框架默认限制在一屏内，不出现浏览器默认滚动条；如果后续某个页面需要滚动，优先放进布局内部内容区处理。
- 登录页、后台布局、空状态页这类强调页面结构和氛围的界面，优先使用这套混合方式。
- 如果某个效果 Naive UI 已经有成熟实现，先用组件能力，不要为了“全 Tailwind”重复造轮子。

::: details 为什么这一章不走“全 Tailwind”或“全组件库”
只用组件库，页面骨架很容易写得保守；只用 Tailwind，又会把表单、按钮、弹窗这些后台高频能力重新手写一遍。

当前项目更适合折中方案：

- Tailwind 4 负责把页面结构写快、写稳、写得更贴近原型。
- Naive UI 负责表单、按钮、消息、弹窗、卡片这类后台基础组件。

这样既能保持开发效率，也能让页面视觉层次更灵活。
:::
