# 部门树与部门管理

角色和用户都已经进入最终结构后，组织体系里最核心的资源就是部门本身。到了这一节，`sys_department` 不再只是为了支持数据权限而存在的底表，而会变成一套真正可管理的组织模块。

::: tip 🎯 本节目标
完成后，部门模块会和用户、角色一样进入最终版结构，并支持部门树查询、创建、编辑、状态切换，以及按当前登录人的数据范围过滤可见部门。
:::

## 本节会改什么

这次新增的是 `iam/department` 模块：

```text
server/
├─ internal/
│  ├─ module/
│  │  ├─ iam/
│  │  │  └─ department/
│  │  │     ├─ datascope.go
│  │  │     ├─ dto.go
│  │  │     ├─ entity.go
│  │  │     ├─ handler.go
│  │  │     ├─ policy.go
│  │  │     ├─ repository.go
│  │  │     ├─ routes.go
│  │  │     └─ service.go
│  │  └─ system/
│  │     └─ routes.go
│  └─ platform/
│     └─ datascope/
│        └─ datascope.go
```

| 文件 | 作用 |
| --- | --- |
| `dto.go` | 部门请求体、响应结构和状态校验 |
| `repository.go` | 部门树查询、编码校验、负责人校验、祖先路径更新 |
| `service.go` | 组织树变更规则、子树祖先路径重写、事务边界 |
| `handler.go` | HTTP 协议层绑定与输出 |
| `datascope.go` | 固定部门资源自身的数据权限接法 |

## 为什么部门模块不能只做平铺 CRUD

部门和普通字典项最大的区别在于，它天生是树形资源，而且这个树形关系还会直接影响数据权限。

也就是说，部门模块至少要同时解决三件事：

- 怎么表达父子层级
- 部门调整上级后，祖先路径怎么保持一致
- 当前登录人到底能看到哪一段部门树

如果这里只做成“能增删改查一张表”的平铺 CRUD，后续部门迁移、子树移动和数据权限都会很快出问题。

## 部门模块现在对外长什么样

当前主线先给部门模块补 4 个接口：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/system/departments` | 查询部门树 |
| `POST` | `/api/v1/system/departments` | 创建部门 |
| `POST` | `/api/v1/system/departments/:id/update` | 编辑部门 |
| `POST` | `/api/v1/system/departments/:id/status` | 修改部门状态 |

这组接口已经挂进系统模块路由，和用户、角色模块走的是同一条聚合链路。

## 部门资源的数据权限怎么接

部门模块和用户模块不一样。  
用户资源是“某个用户属于哪个部门”，而部门资源本身就是“部门节点”。

所以这条主线里，部门模块会单独使用一条资源级作用域：

```go
func applyDataScope(db *gorm.DB, actor datascope.Actor) *gorm.DB {
	return db.Scopes(datascope.DepartmentQueryScope(db, actor, "id"))
}
```

这意味着部门列表会按“部门 ID 本身是否在当前角色可见范围内”过滤，而不是再去看别的归属字段。

::: info 这里有一个当前主线里的约定
当角色数据范围是 `self` 时，部门资源会退化为“当前用户所在部门”。这样至少还能看到自己的组织归属，不会出现部门模块完全空白的情况。
:::

## 祖先路径为什么要在更新时同步改子树

部门树首版采用的是：

- `parent_id`
- `ancestors`

所以一旦把某个部门挂到新的父部门下面，这个部门所有子孙节点的 `ancestors` 都必须跟着更新。

举个例子：

- 原来 `销售一部` 的路径是 `0,1,3`
- 现在把它挪到新的父部门下面

那么不只是 `销售一部` 自己要更新，`销售一部` 的所有下级部门也都要跟着重写祖先路径。

这就是为什么部门模块的 Service 里会专门处理“子树祖先路径重写”，而不是只更新当前节点一行记录。

## 部门管理现在额外会校验什么

除了普通的名称、编码、状态这些字段校验，这一节还会多做几层组织资源特有的保护：

- 不能把部门挂到自己下面
- 不能把部门挂到自己的子部门下面
- 部门编码必须唯一
- 负责人如果存在，必须是启用中的用户

这些规则都属于“组织树一致性”，不属于前端表单能不能提交通常意义上的校验。

## 怎么验证这一节已经做成

### 1. 后端构建通过

在 `server/` 目录执行：

```bash
go test ./...
```

应该能看到 `internal/module/iam/department` 已经进入编译链路，并且整体通过。

### 2. 部门接口已经进入系统路由

登录后请求：

```text
GET /api/v1/system/departments
```

应该能拿到树形结构，而不是平铺列表。

### 3. 部门树会按数据范围裁剪

当前登录人如果不是 `all` 范围，那么：

- 只能看到当前角色允许可见的部门节点
- 更新部门时，也只能操作当前范围内能查到的节点

这说明数据权限已经不仅落在“用户资源”，也开始真正约束组织资源本身了。

### 4. 移动部门时会同步更新子树路径

如果把某个有下级部门的节点改挂到新的父节点下面，子节点的 `ancestors` 也应该一起更新。  
这一点虽然页面上不一定一眼能看到，但它会直接影响后续“本部门及子部门”的查询是否正确。

## 本节最关键的收获

这一节真正要建立的判断标准是：

> 组织资源不是“后台里再加一个管理页”这么简单，它会直接影响数据权限和组织树一致性，所以必须尽早按最终结构落地。

部门模块的价值不只是补一套组织管理接口，而是把“组织树如何长期维护”这件事真正收进底座。

## 下一步

部门模块定稳之后，下一步最自然的延伸就是岗位模块。届时组织体系会从“部门树”扩展到“部门 + 岗位 + 用户归属”的完整闭环。
