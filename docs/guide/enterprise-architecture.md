# 企业级架构升级

::: tip 🎯 这页解决什么
帮你快速理解：为什么 `EZ Admin Gin` 不再只满足于“功能能跑通”，而是开始升级为可长期扩展的企业级单体底座。
:::

## 先看结论

`v1.1.0` 已经是一套完整可用的后台系统，但它更接近“功能型后台”。继续往上叠组织体系、数据权限、复杂查询治理时，如果还沿用原来的结构，成本会越来越高。

所以 `v2` 的第一阶段，不是先加更多页面，而是先做结构升级：

- 启动入口从 `main.go` 的集中装配，升级为 `bootstrap` 聚合
- 技术基础能力收拢到 `platform`
- 路由按模块拆成 `module/auth`、`module/setup`、`module/system`
- 先把部门、岗位、角色数据范围这些基础模型和迁移打底
- 为后续把业务逻辑从 `handler` 迁到 `service / repository` 铺路

这一步的目标不是炫技，而是为了让后续的组织体系和数据权限能真正落地。

## 为什么 v1 结构不够了

当前仓库在 `v1` 阶段已经做对了很多事：

- 配置、日志、数据库、Redis、迁移链路都完整
- JWT、Casbin、菜单、按钮、日志、公告都已跑通
- 前后端和部署文档已经形成闭环

但如果目标升级为企业级底座，下面这些问题会开始放大：

- `handler` 里直接操作 `gorm.DB`，参数绑定、事务、查询拼装和业务规则混在一起
- 当前权限体系只解决“接口能不能进”，还没解决“进去后能看到哪些数据”
- 路由集中在一个文件里注册，模块一多会变成总装配文件
- 组织体系、数据权限和查询规则还没有稳定的落点

这也是为什么 `v2` 要先做结构升级，而不是只继续补模块。

## v2 第一阶段已经落地的骨架

这一阶段先引入新的命名空间和装配方式，但尽量保持现有运行行为不变。

新的后端骨架会逐步朝下面这个目标收敛：

```text
server/
├── cmd/
│   └── server/
├── internal/
│   ├── bootstrap/
│   ├── platform/
│   │   ├── config/
│   │   ├── database/
│   │   ├── logger/
│   │   ├── redis/
│   │   ├── authn/
│   │   ├── authz/
│   │   └── datascope/
│   └── module/
│       ├── auth/
│       ├── iam/
│       ├── org/
│       ├── system/
│       ├── dict/
│       └── account/
```

第一阶段的重点不是把所有历史代码一次搬完，而是先建立好新的“落点”：

- `bootstrap` 负责启动装配和模块聚合
- `platform` 负责技术基础设施
- `module/*/routes.go` 负责模块级路由注册
- `datascope` 先定义企业后台常见的数据权限模型

## 这次升级和数据权限是什么关系

数据权限不是一条额外规则，而是整个企业后台结构是否合理的试金石。

如果系统里只有登录和接口权限，那么：

- 用户能不能访问“用户列表接口”已经够判断权限了

但只要进入企业后台，就会很快遇到第二层问题：

- 这个角色能不能只看本部门用户？
- 能不能看本部门及子部门？
- 能不能看自己创建的数据？
- 多个角色叠加后数据范围怎么合并？

这类问题如果继续堆在 `handler` 里，后续每个模块都会越来越重。  
所以 `v2` 明确把权限拆成两层：

- **接口权限**：决定“能不能访问接口”
- **数据权限**：决定“访问后能看到哪些数据”

接口权限继续用 `Casbin`，数据权限则会通过 `Actor + ScopeResolver + gorm.Scopes(...)` 逐步落地。

## 当前阶段先做什么

`v2` 的第一阶段先完成三件事：

1. 建立新的后端骨架，让后续代码有稳定归位
2. 落组织体系和数据权限的基础模型与迁移
3. 同步补文档入口，让读者知道项目已经开始从 v1 向 v2 迁移

后续阶段再继续做：

- `service / repository` 分层
- 部门、岗位管理模块
- 角色数据范围配置
- 真正的数据过滤接入
- 企业级教程主线重写
