# EZ Admin Gin 执行规划

## 1. 项目定位

**EZ Admin** 是一个面向个人开发者和小型项目的通用后台管理系统底座。

它的目标不是做一个一次性的后台 Demo，而是沉淀一套可长期复用的基础能力。以后每个个人项目、工具站、小型产品原型，都可以基于它快速接入业务模块并上线验证。

当前仓库建议命名为：

```text
ez-admin-gin
```

项目展示名：

```text
EZ Admin
```

一句话定位：

```text
面向个人项目快速上线的 Go/Gin 通用后台管理系统底座。
```

## 2. 核心目标

项目优先解决这些问题：

- 避免每个新项目重复搭建后台系统
- 提供统一的登录、权限、菜单、配置、日志、文件等基础能力
- 支持快速新增业务模块
- 支持本地开发、测试和 Docker 部署
- 形成一套自己长期可维护、可扩展的后台开发模板

## 3. 项目结构规划

推荐采用单仓库结构：

```text
ez-admin-gin/
├── server/          # Go/Gin 后端服务
├── web/             # Vue 3 后台管理前端
├── docs/            # VitePress 文档
├── deploy/          # Docker、Nginx、部署配置
├── scripts/         # 常用脚本
└── README.md
```

后端核心结构：

```text
server/
├── cmd/             # 应用入口
├── internal/
│   ├── system/      # 通用后台能力
│   ├── modules/     # 业务模块
│   ├── middleware/  # 中间件
│   ├── router/      # 路由注册
│   └── bootstrap/   # 启动初始化
├── pkg/             # 可复用工具包
├── configs/         # 配置文件
└── migrations/      # 数据库迁移
```

## 4. 功能边界

### 第一阶段先做通用底座

MVP 范围：

- 登录认证
- 用户管理
- 角色管理
- 权限管理
- 菜单管理
- 系统配置
- 文件上传
- 操作日志
- 基础前端管理台
- Docker Compose 本地部署
- 一个示例业务模块

### 暂缓功能

这些功能后续再做，不进入第一版：

- 多租户
- 工作流
- 插件系统
- 消息通知
- 定时任务平台
- 复杂数据权限
- 微服务拆分

第一版原则是：**先做能复用、能上线、能扩展的底座，不追求大而全。**

## 6. 开发路线

### 阶段一：项目初始化

目标：建立可运行的项目骨架。

任务：

- 创建 GitHub 仓库 `ez-admin-gin`
- 初始化目录结构
- 初始化 Go 后端项目
- 初始化 Vue 前端项目
- 初始化 VitePress 文档
- 准备 Docker Compose 基础环境

### 阶段二：后端基础设施

目标：让后端具备真实项目的基础能力。

任务：

- 配置管理
- 日志系统
- 数据库连接
- Redis 连接
- 统一响应格式
- 错误处理
- 路由分组
- 健康检查接口

### 阶段三：认证与权限

目标：完成后台系统最核心的安全能力。

任务：

- 用户模型
- 登录接口
- JWT 认证
- 认证中间件
- 角色模型
- 权限模型
- Casbin 权限控制
- 菜单权限设计

### 阶段四：通用系统模块

目标：补齐后台底座常用能力。

任务：

- 用户管理
- 角色管理
- 菜单管理
- 系统配置
- 文件上传
- 操作日志
- 登录日志

### 阶段五：前端管理台

目标：让系统可视化可操作。

任务：

- 登录页
- 后台布局
- 动态菜单
- 用户管理页面
- 角色管理页面
- 菜单管理页面
- 系统配置页面
- 文件管理页面

### 阶段六：业务模块接入规范

目标：定义以后小项目如何接入。

每个业务模块建议遵循固定结构：

```text
model
repository
service
handler
router
permission
menu
migration
frontend page
```

先实现一个示例模块，比如：

- 文章管理
- 工具管理
- 资源导航
- 项目管理

示例模块的目的不是功能复杂，而是证明这个底座可以承载真实业务。

### 阶段七：部署与复用

目标：让项目可以快速上线和复制。

任务：

- Dockerfile
- Docker Compose
- Nginx 配置
- 环境变量配置
- 初始化数据脚本
- 部署文档
- 新项目复用说明

## 7. 文档规划

文档不再只写“技术教程”，而是分成三类：

```text
docs/
├── guide/           # 使用指南
├── tutorial/        # 从零搭建教程
├── reference/       # 配置、接口、模块参考
└── roadmap.md       # 路线图
```

文档主线应该围绕：

```text
如何搭建一个以后能持续复用的后台系统底座
```

而不是单纯围绕：

```text
如何学习 Gin
```

## 8. 执行原则

- 先跑通，再优化
- 先做通用能力，再做业务能力
- 先单体架构，不急着拆微服务
- 先保证可维护，不追求复杂设计
- 所有模块都要服务于“以后个人项目快速上线”这个目标
- 文档和代码同步推进，避免只写代码没有使用说明
